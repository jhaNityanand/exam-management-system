Architecture

A conventional Laravel monolith with server-rendered Blade pages, page-specific JavaScript, JSON endpoints for rich lists, and service classes for domain workflows.

Browser→Web routes→Controller + Form Request→Service→Eloquent / Storage→Blade or JSON

Server side

  • Laravel 11.50, requiring PHP 8.2+.
  • Controllers are split into Backend, Frontend, Api\Workspace, and Auth.
  • Form Requests centralize validation for questions, exams, categories, blogs, news, gallery and import operations.
  • Services own domain behavior such as exam creation, category trees, question import, gallery persistence and attempt assignment.
  • Eloquent models use organization scopes, soft deletes, casts and explicit relationships.

Client side

  • Blade templates render public and administration interfaces.
  • Tailwind CSS 3 provides utility styling; larger modules also use scoped files under public/css.
  • Alpine.js powers light interactions; dedicated ES modules handle tables, forms, import and media tools.
  • Axios/fetch communicate with protected internal JSON endpoints.
  • Vite builds resources/css/app.css and resources/js/app.js.

Important directories

  • app/Http/Controllers — HTTP orchestration
  • app/Http/Requests — validation and authorization hooks
  • app/Models — domain and CMS models
  • app/Services — business workflows
  • app/Support — shared options/query helpers
  • database/migrations — schema history
  • database/seeders — complete demo data
  • resources/views/backend — administration UI
  • resources/views/frontend — public website and account UI
  • resources/js/core — theme/sidebar/table foundations
  • public/js/backend — page-level backend scripts
  • public/css/backend — page-level backend styles
  • tests/Feature — end-to-end application behavior

Installation and quick start

Requirements

  • PHP 8.2 or newer with common Laravel extensions (openssl, pdo, mbstring, tokenizer, xml, ctype, json, fileinfo, gd)
  • Composer 2
  • Node.js 18+ and npm
  • MySQL 8 / MariaDB 10.4+ (SQLite is fine for automated tests)
  • A web server such as Apache, Nginx, Laravel’s development server, or WAMP

WAMP / Apache notes

  • Document root must be the project public/ directory (or a vhost alias to it).
  • Set APP_URL to the exact public URL you browse (including port/path if needed).
  • Run php artisan storage:link so gallery files are reachable under /storage.
  • Use npm run build for a one-shot asset build, or keep npm run dev running while editing frontend files.

The queue uses the database driver by default. The tracked question import itself is request-based and does not require a queue worker.

Questions, categories and spreadsheet import

Question bank

  • Organization-scoped create, view, edit, soft delete, restore, bulk status and bulk recycle-bin actions.
  • AJAX list with search, category, type, difficulty, status, source and pagination controls.
  • Supported authoring types include MCQ, true/false, fill blank, short answer and long answer.
  • Question categories are maintained as nested trees with status, ordering and SEO fields.
  • Imported questions expose a clickable source badge linked to their import record.

Tracked import lifecycle

Select XLSX/CSV→Browser preview→Validate/edit rows→Upload source→Import chunks→Finalize log
  • Maximum upload size: 15 MB.
  • Maximum rows: 10,000; the first Excel worksheet is processed.
  • Rows are sent sequentially in batches of 100 to constrain request size and memory.
  • The original source file is stored on the private local disk and can be downloaded through an authenticated, organization-scoped controller action.
  • import_questions tracks file metadata, status, totals, successes, failures, logs, errors, creator and timestamps.
  • questions.import_question_id links each successfully imported question to its source.
  • Every row is validated again on the server. Valid rows remain committed even when other rows fail.

Spreadsheet columns

GroupColumns

Exams and attempt assignment

Exam authoring

  • Create/edit forms share options from App\Support\ExamFormOptions and validation rules from dedicated Form Requests.
  • The live question bank loads categories, counts, filtered questions and random selections through protected JSON routes.
  • Configuration covers duration, attempts, passing rules, visibility, schedule, negative marking, question/option shuffling, paper sets and result behavior.
  • List and detail pages provide analytics, filtering, bulk actions, soft deletion, restoration and publishing.
  • OG title, description and gallery image fields integrate with frontend social metadata.

Question selection modes

Fixed

The author selects the exact questions attached through exam_question. Every candidate receives that set, optionally shuffled.

Pool

A larger attached pool is stored and a stable candidate subset is sampled when the attempt starts.

Dynamic

Questions are selected at attempt time from category rules; the exam pivot remains empty.

Attempt guarantees already implemented

  • Starting an attempt is transactional and resumes an existing active attempt.
  • Assigned rows are stored in exam_attempt_questions with position, marks, option order and selection metadata.
  • Question snapshots are immutable if the source question is edited later.
  • Candidate payloads omit correct answers.
  • Published status, fixed schedule window and attempt limits are validated.
  • Question shortages roll back the attempt and return a structured validation response.
Current boundary: assignment/start is implemented, but the complete candidate workspace—autosave, timer, submit, scoring, pass/fail and result review—still needs implementation.

CMS, SEO and public frontend

SEO model

SEO sections use one canonical slug input and support meta title/description, canonical URL, Open Graph title/description and a reusable gallery-backed OG image picker. OG images are available for questions, exams and all question/exam/blog/news category types; blog and news content also use their existing media integration.

Public experience

  • Database-driven home page sections, navigation and site settings.
  • Published exam, blog, news and category listings with filters and pagination.
  • Global search and JSON search suggestions.
  • Root-level clean CMS page slugs; /pages/{slug} redirects permanently to /{slug}.
  • Contact submissions and newsletter subscriptions.
  • Authenticated account dashboard, exams, results and settings pages.

Gallery and file storage

  • Gallery uploads use GALLERY_DISK (default public) and require php artisan storage:link.
  • Supported kinds are images, videos and documents, each with configurable size and extension rules.
  • Rows preserve original and optional modified paths; reverting removes only the modified asset.
  • Image metadata includes dimensions and generated thumbnail paths.
  • Search, kind/source/folder filters, sorting, pagination and statistics are exposed through gallery endpoints.
  • Delete moves records into a bin; restore and force-delete actions are separate.
  • Rich-text editor uploads use the same gallery service and organization ownership.
Question import files intentionally do not use the public gallery. They remain on the private local disk because spreadsheets may contain sensitive question-bank content.

Database and domain model

The schema is migration-driven and uses organization foreign keys throughout authoring modules. Major entities are grouped below.

Identity and tenancy

users, profiles, organizations, user_organizations, user_app_settings, system_settings.

Question bank

questions, question_categories, import_questions.

Exam engine

exams, exam_categories, exam_question, exam_attempts, exam_attempt_questions, instruction rules and templates.

Publishing

blogs, blog categories/tags/banners/attachments and equivalent news entities.

Media

galleries stores organization ownership, paths, URLs, metadata, edit state, thumbnails and audit columns.

Frontend CMS

Site settings/pages/menus, hero banners, FAQs, testimonials, partners, ads, contact messages, social links and newsletter subscribers.

Common conventions

  • Most authoring records include organization_id and are queried with forOrg().
  • Recoverable content uses Eloquent soft deletes.
  • JSON casts store structured options, answers, configuration, logs and snapshots.
  • Foreign keys use cascade or null-on-delete behavior appropriate to ownership and optional media links.
  • Audit columns such as created_by and updated_by identify actors where supported.

UI, assets and theme system

  • Tailwind is configured with darkMode: 'class'.
  • resources/views/partials/theme-init.blade.php applies the theme before CSS paints to reduce flash.
  • resources/js/core/theme.js resolves light/dark/system preference and stores it under ems.theme.
  • Resolution order is local storage, server-provided data-theme-default, then OS preference for system mode.
  • The shared <x-theme-toggle /> component is used in backend, guest and public layouts.
  • Complex page CSS uses .dark overrides or theme-aware variables; third-party editors and dialogs require explicit dark styles.
  • versioned_asset() adds file modification timestamps to public assets for cache busting.

Development guide

Adding a backend module

  1. Create migration and organization-scoped Eloquent model.
  2. Add Form Requests with validation and organization-aware exists rules.
  3. Place domain mutations and complex queries in a service.
  4. Keep controllers focused on organization resolution, orchestration and responses.
  5. Add routes inside the authenticated admin group, before dynamic catch-all routes.
  6. Build Blade screens using existing panel, form, gallery-picker and list conventions.
  7. Add feature tests for authentication, organization isolation, CRUD, validation and lifecycle actions.

Existing reusable patterns

  • ResolvesCurrentOrganization — controller organization resolution.
  • BelongsToOrganization — model organization scope.
  • HierarchicalCategoryService — nested category persistence.
  • HandlesCategoryListActions — shared bulk list actions.
  • ajax-table.js and workspace data controllers — server-driven tables.
  • gallery-picker.blade.php — reusable single/multiple media selection.
  • seo-og-image.blade.php — consistent Open Graph image field.
  • ExamFormOptions — one source for exam option lists.

Production checklist

  1. Set APP_ENV=production, APP_DEBUG=false, the canonical APP_URL, and a secure generated key.
  2. Configure production database, mail, queue, cache and session drivers.
  3. Install optimized PHP dependencies with composer install --no-dev --optimize-autoloader.
  4. Build assets with npm ci && npm run build.
  5. Run php artisan migrate --force. Do not run demo seeders unless explicitly required.
  6. Run php artisan storage:link when the public gallery disk is used.
  7. Cache framework metadata with php artisan optimize.
  8. Run a persistent queue worker for queued jobs and restart it after deployments.
  9. Point the web server document root to public/; deny direct access to .env, storage internals and source files.
  10. Back up the database and both public gallery and private import storage.

Security and current constraints

Authorization gap: administration routes require authentication but do not yet have route-level role middleware or model policies. Organization checks protect many domain records, but role labels alone are not an authorization boundary.
  • Form Requests validate uploaded files, enums, relationships and organization-owned gallery IDs.
  • Controllers and services commonly scope records by the resolved organization and return 403/404 for cross-organization access.
  • Authentication uses Laravel sessions, CSRF protection, login throttling and signed email verification links.
  • Question import source files are private and downloaded only through application routes.
  • Chunked import requests validate import_question_id against the current organization and processing status.
  • Gallery validation applies type and size allowlists, but production web-server MIME and malware scanning policies should be added as needed.
  • The application is effectively single-active-organization: the helper resolves an authenticated user’s first active membership, while shared navigation currently displays the first organization. A validated session switcher is future work.
  • Public registration remains enabled by Breeze.

Troubleshooting

SymptomResolution
Uploaded gallery images return 404Run php artisan storage:link; verify APP_URL, GALLERY_DISK and filesystem permissions.
Vite manifest missingRun npm install and npm run build, or keep npm run dev active.
Database table/session/cache errorsCheck DB credentials, create the database, then run php artisan migrate.
CSS or Blade changes appear staleRun php artisan optimize:clear and rebuild/watch assets as applicable.
Email is not received locallyThe default mailer writes messages to logs. Inspect Laravel logs or configure SMTP.
Queue work does not runStart php artisan queue:work and ensure the jobs table migration is applied.
Import stops after some chunksCompleted chunks remain stored. Review the import details/errors, correct failed rows and retry only those rows.
OG image picker looks unstyled or does not openConfirm public/css/backend/gallery-picker.css and public/js/backend/content-form-shared.js load on the page; hard-refresh after asset changes.
No organization can be resolvedRun the organization/user membership seeders or create an organization and active membership.
